# 第一次落地修改：如何安全地读、改、验证

## 这篇文档解决什么问题

前面 8 篇里，你已经建立了这个项目的大图景。

现在要解决最后一个问题：

> 如果你第一次真的要改 AionUi，应该从哪里下手？
> 怎样避免一上来就把自己绕进复杂度里？

这一篇不是讲某个功能本身，而是讲**落地方法**。

目标是让你从“能看懂”走到“敢开始做第一个改动”。

---

## 先给结论：第一次不要直接改“大功能”

如果你刚理解这个项目，不要一上来就碰：

- 扩展系统整体设计
- Team 多 agent 调度策略
- WebUI 认证深层机制
- Channel 插件接入

第一次落地，应该选这种链路：

1. UI 可见
2. 后台可追踪
3. 运行时边界清晰
4. 有现成测试或容易补测试

从当前代码看，最适合初学者的两个练手入口是：

### 入口 A：Team 创建链路

- `src/renderer/pages/team/components/TeamCreateModal.tsx`
- `src/process/bridge/teamBridge.ts`
- `src/process/team/TeamSessionService.ts`

### 入口 B：WebUI 密码修改链路

- `src/renderer/components/settings/SettingsModal/contents/WebuiModalContent.tsx`
- `src/process/bridge/webuiBridge.ts`

这两条链路都具备：

- UI 按钮 / 表单明显
- bridge 中转清晰
- 后台落点清楚
- 易于验证

---

## 一、第一次改动前，先做 4 个定位动作

### 1. 先定运行时边界

先问自己：

- 这是 `renderer` 问题？
- 这是 `process` 问题？
- 这是 `worker` 问题？
- 这是 `common` 共享契约问题？

如果你不先分清这个，后面很容易把前端问题改进主进程，或者把主进程逻辑误塞进 renderer。

### 2. 再定真实入口

不要凭感觉猜入口，先用代码事实锁定：

- UI 从哪个组件触发？
- 它调的是哪个 `ipcBridge.xxx.invoke(...)`？
- 对应的 provider 在哪个 `bridge` 文件？
- 真正业务实现在哪个 `service` / `team` / `task` 文件？

### 3. 再定验证层级

问自己：

- 这更适合 unit test 吗？
- 需要 integration test 吗？
- 是不是要补一个 E2E 观察点？

### 4. 最后才开始改代码

AionUi 这种分层明显的项目里，“先动手再理解”很容易白改。

---

## 二、推荐的第一次实战路径：Team 创建

### 第一步：从 UI 组件开始

看 `TeamCreateModal.tsx`，你要先读懂：

- 输入项有哪些
- 点击 Create 时构造了什么参数
- 调用了哪个 bridge

这里最重要的一行是：

```ts
ipcBridge.team.create.invoke(...)
```

### 第二步：追到 `teamBridge.ts`

确认：

- 谁注册了 `team.create.provider(...)`
- 异常如何返回给前端

### 第三步：追到 `TeamSessionService.createTeam(...)`

确认：

- teamId 怎么生成
- agent conversation 怎么创建 / 复用
- workspace 怎么回填
- team 结构什么时候落库

### 第四步：如果改动影响 team 启动，再看 `getOrStartSession(...)`

因为有些 Team 相关问题不是创建时出现，而是 session 真正启动时出现。

---

## 三、推荐的第二条实战路径：WebUI 密码修改

### 第一步：看前端分叉

`WebuiModalContent.tsx` 里先看：

- 桌面环境为什么优先走 `window.electronAPI.webuiChangePassword(...)`
- 浏览器环境为什么改走 `webui.changePassword.invoke(...)`

### 第二步：看后台桥接

在 `webuiBridge.ts` 确认：

- provider 是怎么注册的
- direct IPC 是怎么注册的
- 返回结构长什么样

### 第三步：确认 UI 错误处理和后端错误码是否一致

这是很适合新手做的小改动类型：

- 错误文案不一致
- 密码校验提示不清楚
- 成功 / 失败状态回显不完整

这类改动既能练 bridge，也不会一上来就碰最复杂的 Team 调度。

---

## 四、第一次改动时，建议你固定用这套追踪模板

每次开始一个改动，都可以按下面 6 步走：

### 模板

1. **用户动作是什么？**
   - 点按钮、输文本、切换设置？
2. **前端入口在哪？**
   - 哪个 `.tsx` 文件？
3. **调用了哪个 bridge？**
   - 哪个 `ipcBridge.xxx`？
4. **后台 provider 在哪？**
   - 哪个 `src/process/bridge/*.ts`？
5. **真正业务落点在哪？**
   - 哪个 service / session / task manager？
6. **验证应该怎么做？**
   - unit / integration / e2e / 构建 / i18n？

只要你按这个模板走，基本不会迷路。

---

## 五、改动时最常见的 6 个坑

### 1. 直接在 renderer 里想改完业务

AionUi 很多真正逻辑都不在前端。只改 `.tsx` 经常只是改了表面。

### 2. 忘记区分桌面与 WebUI

有些功能桌面和 WebUI 入口不一样，不能只测一个。

### 3. 把 conversation 和 task 混为一谈

conversation 是持久态，task 是运行态。很多 bug 就是卡在这里。

### 4. 看到 Team 只改 UI，不看 session

Team 很多行为在 `TeamSessionService`、`TeamSession`、`TeammateManager`。

### 5. 只跑页面，不跑测试

本仓库的复杂度决定了你不能只靠“点一下感觉没坏”。

### 6. 凭记忆跑命令

命令要以 `package.json` / `justfile` 为准。

---

## 六、第一次改动后的验证顺序

建议固定按下面顺序来：

### 1. 先看变更范围

确认自己只改了预期文件。

### 2. 再跑最贴近的测试

例如：

- 改 service → 优先 unit / integration
- 改 Team 交互 → 看 team 集成测试 / e2e
- 改 WebUI 设置 → 看 webui.e2e.ts

### 3. 再跑基础质量命令

至少确认：

- lint / format
- 需要时跑 type check

### 4. 跨层变更再补纵向验证

尤其是：

- Team
- WebUI
- 扩展系统
- conversation send path

---

## 七、作为初学者，你第一次最该学会的不是“写代码”，而是“守边界”

在这个项目里，真正的进阶不是写出一段逻辑，而是知道：

- 什么该放 renderer
- 什么该放 process
- 什么该放 bridge
- 什么该交给 task manager
- 什么要用测试证明

一旦这层边界感建立起来，你以后看陌生模块也会快很多。

---

## 八、给你的第一轮实战建议

如果现在就要开始动手，我建议你按这个顺序练：

1. 读 `TeamCreateModal.tsx` → `teamBridge.ts` → `TeamSessionService.ts`
2. 读 `WebuiModalContent.tsx` → `webuiBridge.ts`
3. 选一个非常小的改动点
   - 文案修正
   - 表单校验提示
   - 成功 / 失败状态提示
   - 默认值显示问题
4. 跑对应测试 / 至少做链路验证
5. 再逐步碰更深层逻辑

不要一开始就追求“做大功能”。

## 容易误解的点

- **误解：第一次就应该从最核心模块下手。**
  - 不对。第一次应该选边界清晰的小链路。
- **误解：改动成功就是页面看起来正常。**
  - 不对。你还要看测试与工程验证。
- **误解：理解项目 = 理解目录。**
  - 不对。真正的理解是能沿着用户动作把链路追到底。
