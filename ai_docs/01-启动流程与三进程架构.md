# 启动流程与三进程架构

## 这篇文档解决什么问题

这一篇只解决一个核心问题：

> AionUi 从启动到可用，究竟有哪些运行角色？它们怎么协作？

如果你不先搞清楚这一点，后面读任何业务代码都会混乱。

---

## 先给结论：AionUi 不是“三层页面”，而是“多运行时系统”

从当前代码看，AionUi 至少有 5 个运行角色：

1. **Main process（主进程）**
2. **Preload（预加载桥）**
3. **Renderer（渲染进程 / UI）**
4. **Worker / Agent runtime（后台执行单元）**
5. **Standalone Web server（独立 Web 服务端）**

其中前 3 个是 Electron 桌面应用的核心，后 2 个是为了 AI 执行和 WebUI 模式扩展出来的能力。

你可以先记这个图：

```text
桌面模式
用户 -> Renderer -> Preload -> Main -> Services / Bridge / Task / Team / Extensions
                                           |
                                           -> Worker / Agent runtime

WebUI 模式
浏览器 -> WebSocket / HTTP -> Standalone server -> Bridge / Services / Task / Team
```

---

## 一、桌面模式的启动主线

桌面模式的总入口是 `src/index.ts`。

从 `handleAppReady()` 可以看到，桌面启动的大致顺序是：

1. Electron 完成 `app.whenReady()`
2. 执行 `initializeProcess()`
3. 恢复缩放、托盘等桌面设置
4. 创建主窗口 `createWindow()`
5. 初始化 ACP 检测器
6. 初始化 i18n（国际化）与托盘文案
7. 异步恢复 WebUI、Pet、Shell 环境等附加能力

### 关键观察

#### 1. `initializeProcess()` 在窗口创建前执行

这是很重要的设计信号。

说明这个项目把很多能力都当成**后台基础设施**，而不是“等 UI 打开后再懒加载”。

`src/process/index.ts` 里它做了三件最关键的事：

- `initStorage()`
- `ExtensionRegistry.getInstance().initialize()`
- `getChannelManager().initialize()`

也就是说：

- 存储层要先准备好；
- 扩展系统要先扫描和解析好；
- 渠道系统要先就绪；
- 然后 UI 才开始承载用户操作。

#### 2. 窗口本身只是“壳”

`createWindow()` 负责：

- BrowserWindow 创建
- 标题栏样式
- preload 注入
- renderer 加载 URL / HTML
- DevTools 状态广播
- 托盘关闭拦截

它不负责业务初始化。

这意味着：

> 业务能力主要不在窗口里，而在 `src/process/`。

---

## 二、Main process 到底负责什么

你可以把 `src/process/` 理解成“桌面后端”。

它不是浏览器端逻辑，而是运行在 Node.js / Electron 主进程里的后台层。

### 当前主进程最关键的 6 个子域

| 目录 | 作用 |
| --- | --- |
| `src/process/bridge/` | 暴露给 renderer / WebUI 的业务入口 |
| `src/process/services/` | 会话、数据库、更新、预览历史等后台服务 |
| `src/process/task/` | agent runtime 和任务生命周期管理 |
| `src/process/team/` | Team 多 agent 协作层 |
| `src/process/extensions/` | 扩展发现、激活、贡献解析 |
| `src/process/channels/` | Telegram / 钉钉 / 微信 / 企微等外部渠道 |

这几个目录拼起来，才是 AionUi 的真正“应用内核”。

---

## 三、Preload 为什么单独存在

`src/preload/main.ts` 的角色非常明确：

- 用 `contextBridge.exposeInMainWorld('electronAPI', ...)` 暴露一个**最小可用**的桌面 API 面
- 让 renderer 不直接碰 `ipcRenderer` 之外更危险的 Electron / Node 能力

从代码能看到，它暴露的主要能力有：

- 通用 `emit` / `on`
- 文件拖拽路径获取
- WebUI 直连 IPC（如改密码、查状态）
- 微信登录相关 IPC
- 反馈日志收集

你可以把 preload 看成：

> Renderer 与 Main 之间的“安全前门”。

它存在的意义不是增加复杂度，而是防止 renderer 直接拥有 Node.js 权限。

---

## 四、Renderer 在整个架构里只是 UI 层

`src/renderer/main.tsx` 做的事情很纯：

- 初始化 Sentry（仅 Electron 环境）
- 注册 browser adapter
- 挂载 React Provider 树
- 挂载 Router 和 Layout
- 注册 PWA

它主要负责：

- 页面
- 路由
- 状态上下文
- 视觉交互

它**不负责**：

- 数据库存取
- AI agent 启动
- 渠道系统
- Team session 生命周期
- WebUI server 生命周期

这些都在 `process/`。

所以如果你看到 renderer 页面里调用 `ipcBridge.xxx.invoke(...)`，要第一时间意识到：

> 真正的业务逻辑在后面，不在这个组件里。

---

## 五、Worker 为什么是单独的运行域

从 `electron.vite.config.ts` 和 `workerTaskManagerSingleton.ts` 能确认，AionUi 把 AI 执行运行时单独抽出来了。

`workerTaskManagerSingleton.ts` 注册了多种 agent manager：

- `GeminiAgentManager`
- `AcpAgentManager`
- `OpenClawAgentManager`
- `NanoBotAgentManager`
- `RemoteAgentManager`
- `AionrsManager`

而 `WorkerTaskManager` 负责：

- 懒构建 agent task
- 缓存已运行 task
- 替换旧 task 时先 kill
- 空闲超时清理 CLI agent
- 清空所有 task

这说明项目的一个核心设计理念是：

> UI 不直接持有 AI 执行状态，主进程用 task manager 统一管理运行中的 agent。

---

## 六、为什么还有 `src/server.ts`

`src/server.ts` 不是桌面入口，它是**独立 WebUI 模式入口**。

它会：

1. 注册 Node 平台服务
2. 注册 standalone adapter
3. 调用 `initBridgeStandalone()`
4. 启动 `startWebServerWithInstance()`

这里很关键的一点是：

- 桌面模式走 `src/common/adapter/main.ts`
- 独立 Web 模式走 `src/common/adapter/standalone.ts`

两者 transport（传输方式）不同，但后面挂的业务桥接能力高度复用。

这就是 AionUi “双运行时共用一套业务语义”的基础。

---

## 七、这套架构为什么这样分

从代码组织反推，背后的设计意图很清楚。

### 1. 把 UI 与后台能力硬隔离

这样 renderer 可以保持前端心智模型，不必直接持有 Node / Electron 权限。

### 2. 把 transport 和业务分离

桌面走 IPC，WebUI 走 WebSocket / HTTP，但业务桥接和服务层尽量复用。

### 3. 把 AI 运行时独立出来

这样 agent 可以有自己的生命周期、缓存、超时回收、并发策略，不会跟 UI 状态混在一起。

### 4. 让平台能力先启动，再让 UI 挂载

这能让扩展、渠道、配置、桥接接口在窗口显示前就稳定可用。

---

## 初学者应该怎么记这一篇

你不用背所有细节，只要先记住这 4 句话：

1. `src/index.ts` 负责桌面应用编排，不负责大部分业务实现。
2. `src/process/` 才是后台内核。
3. `src/preload/` 是安全桥，不是业务层。
4. `src/renderer/` 是 UI 壳，很多动作都会继续往 `process/bridge/` 走。

## 容易误解的点

- **误解：Preload 是业务层。**
  - 不是。它主要是安全暴露层。
- **误解：WebUI 是另一个前端项目。**
  - 不是。它更多是另一种入口与传输方式。
- **误解：Worker 是前端线程。**
  - 不是。这里指的是后台 AI 运行时管理单元。
