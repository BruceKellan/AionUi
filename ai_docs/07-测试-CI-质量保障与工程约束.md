# 测试、CI、质量保障与工程约束

## 这篇文档解决什么问题

前面你已经理解了：

- 项目的分层与运行时；
- bridge 通信；
- conversation 主链路；
- Team 与 WebUI。

接下来最关键的问题是：

> 这个项目这么复杂，怎么保证它没有坏？
> 当你以后要改代码时，应该用什么工程约束来约束自己？

这一篇就是把“系统认知”落到“工程验证”。

---

## 先给结论

AionUi 的质量保障不是只靠一种测试，而是多层组合：

1. **静态约束**：TypeScript、路径别名、lint、format
2. **单元测试**：Vitest node / jsdom 双项目
3. **集成测试**：真实模块协作验证
4. **E2E 测试**：Playwright 驱动完整流程
5. **CI 工作流**：代码质量、单测、覆盖率、i18n、跨平台构建烟测

这说明它不是“只要能跑起来就行”的项目，而是有明显工程化要求的仓库。

---

## 一、最基础的工程约束：TypeScript 与路径别名

`tsconfig.json` 提供了几个非常重要的约束：

- `noImplicitAny: true`
- `noEmit: true`
- 路径别名：
  - `@/*`
  - `@process/*`
  - `@renderer/*`
  - `@worker/*`

这意味着：

### 1. 代码组织是强依赖分层别名的

如果你不理解：

- `@process` 是主进程代码
- `@renderer` 是前端代码
- `@worker` 是后台执行单元

那你在读 import 时会非常吃力。

### 2. 类型不是可选项

`noImplicitAny` 说明项目希望 API 边界和共享模型尽量明确。

## 二、日常开发命令不是靠猜，而是写死在 `package.json` 与 `justfile`

从 `package.json` 和 `justfile` 可以确认，你在本地常用的命令大致分成几类：

### 开发启动

- `bun run start`
- `bun run webui`
- `just dev`
- `just webui`

### 质量检查

- `bun run lint`
- `bun run lint:fix`
- `bun run format`
- `bun run format:check`
- `bun run i18n:types`

### 测试

- `bun run test`
- `bun run test:coverage`
- `bun run test:integration`
- `bun run test:e2e`

### 构建

- `bun run build`
- `node scripts/build-with-builder.js ...`
- `just build`

所以以后你要验证改动，第一反应应该是看 `package.json` / `justfile`，不是自己发明命令。

---

## 三、Vitest 不是单一测试环境，而是 node / dom 双项目

`vitest.config.ts` 很关键。它明确把测试拆成两类：

### 1. Node 环境

匹配：

- `tests/unit/**/*.test.ts`
- `tests/integration/**/*.test.ts`
- `tests/regression/**/*.test.ts`

这类主要适合：

- 主进程逻辑
- service
- repository
- team runtime
- bridge 层业务

### 2. jsdom 环境

匹配：

- `tests/unit/**/*.dom.test.ts`
- `tests/unit/**/*.dom.test.tsx`

这类主要适合：

- React 组件
- hooks
- 前端交互

这一点非常重要，因为它直接告诉你：

> AionUi 不是“所有单测都跑在浏览器模拟环境”，也不是“所有逻辑都跑在 Node”。

它明确按运行时边界拆了测试环境。

---

## 四、测试 setup 也体现了运行时意识

### `tests/vitest.setup.ts`

这里注册了 `NodePlatformServices`，让依赖平台服务的模块在 Node 测试里能跑。

### `tests/vitest.dom.setup.ts`

这里 mock（模拟）了：

- `electronAPI`
- `ResizeObserver`
- `IntersectionObserver`
- `requestAnimationFrame`
- `localStorage`

这说明前端测试不是直接在真实 Electron 里跑，而是：

> 用 jsdom 模拟浏览器环境，再补齐组件依赖的宿主能力。

---

## 五、单元测试与集成测试的边界很清楚

### 例子 1：`ConversationServiceImpl.test.ts`

这类测试会：

- mock `initAgent`
- mock cron service
- 构造 fake repository
- 直接验证 `ConversationServiceImpl` 的行为

这属于典型 **unit test（单元测试）**。

它关注：

- 业务逻辑是否正确
- merge / migrate / delete 行为是否符合预期

### 例子 2：`team-real-components.test.ts`

这个文件的注释写得很清楚：

- 只 mock 绝对无法在 Node 环境实例化的东西
- Mailbox / TaskManager / TeammateManager / TeamMcpServer 都用真实实现

这属于更靠近 **integration test（集成测试）** 的方式。

它测试的是：

> 多个真实模块接起来以后，系统能否按预期协作。

这类测试对理解 Team 特别有帮助，因为它直接证明 Team 不是纸上架构。

---

## 六、E2E 测试不是装饰，而是覆盖真实关键路径

`playwright.config.ts` 说明了几个关键事实：

- 测试目录是 `tests/e2e/specs`
- `workers: 1`
- `fullyParallel: false`
- Electron tests share one app instance

也就是说：

> E2E 并不是“页面截图玩具”，而是在真实共享 Electron 应用实例上跑的系统验证。

### 当前 E2E 覆盖的真实重点

从 `tests/e2e/specs/` 可以看到，它覆盖了很多主路径，例如：

- app-launch
- conversation-full-cycle
- cron-crud
- team-create
- team-communication
- team-agent-lifecycle
- webui
- extension 相关大批用例

这说明仓库对这些能力是有真实回归要求的。

---

## 七、CI 不只是跑测试，还会做多平台构建烟测

`.github/workflows/pr-checks.yml` 的工作流非常重要。

它大致包含：

### 1. Code Quality

- checkout
- 安装依赖
- 跑 `prek`

这里主要做：

- lint
- format 检查
- 基础代码质量校验

### 2. Unit Tests

在：

- ubuntu
- macos
- windows

三平台跑单测。

### 3. Coverage Test

Linux 上跑 `bun run test:coverage`。

### 4. i18n Check

运行 `node scripts/check-i18n.js`。

### 5. Build Test

还会跑跨平台构建，甚至包括：

- macOS arm64 / x64
- Windows x64 / arm64
- Linux

并做 build artifact（构建产物）验证和部分 smoke test（烟测）。

这说明：

> 对 AionUi 来说，“能过单测”还不够，跨平台构建也是质量门槛的一部分。

---

## 八、从这些测试配置里，你该学到什么

### 1. 改主进程逻辑，不要只看前端页面

你应该优先补：

- unit test
- integration test

### 2. 改 renderer 组件，不要忽略 dom test

很多前端行为在 jsdom 层就能先卡住错误。

### 3. 改团队、WebUI、扩展系统，要特别关注 E2E

因为这些能力跨层很多，单元测试不一定覆盖得住。

### 4. 改文案、设置、前端路由时，i18n 与 build 也会受影响

所以验证不能只停在“本地页面能打开”。

---

## 九、工程约束不只是“怎么测”，还包括“怎么组织代码”

结合 `tsconfig.json`、测试配置和现有目录结构，你以后第一次改代码时最好遵守这几个原则：

1. **先确认运行时边界**
   - 这是 renderer、process、worker、还是 common？
2. **先确认对应测试层级**
   - unit、integration、还是 e2e？
3. **尽量改现有链路，不要平地新造抽象**
4. **引用真实命令，不要凭经验乱跑**
5. **如果是跨层改动，至少做一条纵向链路验证**

## 容易误解的点

- **误解：Vitest 就是一套测试。**
  - 不是。这里明确分了 node 和 jsdom。
- **误解：CI 主要检查代码风格。**
  - 不是。它还检查 coverage、i18n、跨平台 build。
- **误解：E2E 只是补充。**
  - 对 Team、WebUI、扩展这类能力来说，E2E 很关键。
