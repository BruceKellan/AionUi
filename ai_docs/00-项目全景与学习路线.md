# AionUi 项目全景与学习路线

## 这篇文档解决什么问题

如果你现在对 AionUi 完全陌生，这一篇先帮你建立**全局地图**。

你读完后，应该先明确 4 件事：

1. AionUi 到底是什么项目。
2. 它不是单一网页，而是一个多运行时系统。
3. 代码里最重要的几层分别做什么。
4. 你接下来应该按什么顺序学，才不会一上来就迷路。

---

## 一句话理解 AionUi

`package.json` 把 AionUi 描述为：

> Transform your command-line AI agent into a modern, efficient AI Chat interface.

但从当前代码看，它已经不只是“聊天界面”了，而是一个**以 AI 会话为核心、同时支持桌面应用、WebUI、团队协作、扩展系统和多消息渠道接入**的平台。

你可以先把它理解成：

- **表层**：一个 React（前端框架）界面；
- **中层**：一套统一的 bridge（桥接）接口，把界面请求送到主进程；
- **底层**：会话服务、任务管理器、扩展系统、渠道系统、Web server（Web 服务器）等后台能力；
- **AI 运行层**：不同 agent（智能体）后端和 worker（后台执行单元）。

---

## 先记住：AionUi 不是“一个前端项目”

从 `electron.vite.config.ts` 可以确认，这个仓库至少同时构建 3 组入口：

- **main**：`src/index.ts`
- **preload**：`src/preload/main.ts`
- **renderer**：`src/renderer/index.html` + `src/renderer/main.tsx`

并且还额外构建：

- **worker 入口**：`src/process/worker/gemini.ts`、`src/process/extensions/lifecycle/lifecycleRunner.ts`
- **独立 WebUI / 服务端入口**：`src/server.ts`

所以你面对的是一个 **Electron（桌面壳）+ React（渲染界面）+ Node.js（主进程 / 服务端）+ Worker（后台执行）** 的混合系统。

这就是为什么如果你只盯着 `src/renderer/`，很快就会看不懂。

## 当前项目的 5 个核心能力块

### 1. 桌面壳与启动编排

入口在 `src/index.ts`。

它负责：

- Electron 应用生命周期
- 单实例锁
- 主窗口创建
- 自定义协议 `aion-asset://`
- 托盘、菜单、缩放、自动更新
- 桌面模式与 WebUI 模式分流

### 2. 进程内后台能力

入口在 `src/process/index.ts`。

它启动的不是 UI，而是后台基础设施：

- storage（存储）初始化
- `ExtensionRegistry`（扩展注册表）初始化
- `ChannelManager`（渠道管理器）初始化

### 3. bridge 通信中枢

关键文件：

- `src/process/utils/initBridge.ts`
- `src/process/bridge/index.ts`
- `src/common/adapter/main.ts`
- `src/common/adapter/browser.ts`
- `src/preload/main.ts`

这层是整个项目最重要的骨架之一。它解决的问题是：

> Renderer 怎么调用 Main？
> WebUI 又怎么复用同一套业务能力？

### 4. 会话与 AI 运行时

关键目录：

- `src/process/services/`
- `src/process/task/`
- `src/process/team/`

这一层是真正“干活”的地方：

- conversation（会话）怎么创建
- agent（智能体）怎么启动
- workerTaskManager（任务管理器）怎么缓存与回收运行中的 agent
- Team（团队模式）怎么协调多个 agent

### 5. 扩展与外部接入

关键目录：

- `src/process/extensions/`
- `src/process/channels/`

这里说明 AionUi 不是封闭产品，而是有明显的平台化方向：

- extensions（扩展）可以贡献 assistant、skill、theme、MCP server、settings tab、WebUI 能力
- channels（渠道）可以把 Telegram、钉钉、微信、企微等接进来

## 顶层目录怎么读

你不需要一开始读所有目录。先把顶层目录分成 7 类就够了。

| 目录 | 你现在怎么理解它 |
| --- | --- |
| `src/` | 主代码。你主要会在这里学习。 |
| `tests/` | 用来反向验证“项目到底怎么运行”。 |
| `scripts/` | 构建、校验、生成脚本。 |
| `resources/` | 打包资源、图标等。 |
| `public/` | renderer 静态资源。 |
| `docs/` | 已有专题文档，不等于系统化入门路径。 |
| `mobile/` | 独立子域，不是当前主学习主线。 |

在 `src/` 里，再先分 6 个面：

| 目录 / 文件 | 职责 |
| --- | --- |
| `src/index.ts` | Electron 主入口 |
| `src/process/` | 主进程后台能力 |
| `src/preload/` | 安全桥层 |
| `src/renderer/` | React 界面 |
| `src/common/` | 主进程与渲染层共享契约 |
| `src/server.ts` | 纯 WebUI / 独立服务端入口 |

## 你应该如何从 0 学到 100

### 阶段 1：先建立世界观

先看这 4 个文件：

1. `package.json`
2. `electron.vite.config.ts`
3. `src/index.ts`
4. `src/renderer/main.tsx`

目标不是看细节，而是回答：

- 项目有哪些运行模式？
- 从哪里启动？
- 哪些入口属于 main，哪些属于 renderer？

### 阶段 2：理解分层与边界

再看：

1. `src/process/index.ts`
2. `src/preload/main.ts`
3. `src/common/adapter/main.ts`
4. `src/common/adapter/browser.ts`
5. `src/process/bridge/index.ts`

你的目标是看懂一句话：

> 请求是怎么从 UI 进入后台，再把结果送回 UI 的？

### 阶段 3：顺着一条真实业务链路走

推荐先走两条：

1. **Team 创建链路**
   - `src/renderer/pages/team/components/TeamCreateModal.tsx`
   - `src/process/bridge/teamBridge.ts`
   - `src/process/team/TeamSessionService.ts`

2. **WebUI 改密码链路**
   - `src/renderer/components/settings/SettingsModal/contents/WebuiModalContent.tsx`
   - `src/process/bridge/webuiBridge.ts`

这样你会第一次真正感受到：

- 前端按钮怎么触发后台逻辑
- bridge 如何做 transport（传输层）转换
- service 如何落到真正的数据与运行时

### 阶段 4：最后再看工程化

等你已经能跟住链路，再去看：

- `vitest.config.ts`
- `.github/workflows/pr-checks.yml`
- `tests/e2e/specs/team-create.e2e.ts`
- `tests/e2e/specs/webui.e2e.ts`

因为这时你已经知道“系统长什么样”，再看测试和 CI（持续集成）才不会抽象。

## 初学者最容易误解的 4 件事

### 误解 1：`src/renderer/` 就是全部

不是。`renderer` 只是界面壳。大部分关键行为最终都要走 `bridge`，再进 `process`。

### 误解 2：WebUI 是另一套系统

不是。WebUI 复用了大量相同业务语义，只是把 Electron IPC（进程通信）换成了 WebSocket（长连接）/ HTTP。

### 误解 3：Team 是一个前端页面功能

不是。Team 背后有独立的 session、mailbox、task manager、MCP server 协调层。

### 误解 4：扩展系统只是“插件目录”

不是。`ExtensionRegistry` 会扫描、校验、激活、解析贡献项，它更接近一个平台内核。

## 这一篇的核心结论

AionUi 的正确打开方式不是“先看组件”，而是：

1. 先看运行形态；
2. 再看进程边界；
3. 再看 bridge；
4. 再走真实业务链路；
5. 最后再看测试与 CI。

按这个顺序学，你不会在一开始就被细节淹没。
